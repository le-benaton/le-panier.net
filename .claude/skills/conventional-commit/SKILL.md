# Conventional Commit スキル

このプロジェクトのコミット規約に従い、Conventional Commits 形式でコミットを作成する。

## コミットフォーマット

```
<type>(): <日本語で説明>
```

**規約:**
- スコープは常に空 `()` — 省略も内容の記入も不可
- 説明は必ず日本語
- 例: `feat(): メインビジュアルにクロスフェードアニメーションを実装`

## ワークフロー

### Step 1: 変更を確認する

```bash
git status
git diff HEAD
```

ステージング済みとそうでない変更の両方を把握する。

### Step 2: コミットタイプを判断する

変更内容から適切なタイプを選択する。詳細なガイドは `references/commit-types.md` を参照。

**クイックリファレンス:**

| タイプ | 使用場面 |
|--------|----------|
| `feat` | 新機能・新しい UI 要素の追加 |
| `fix` | バグ修正・誤動作の修正 |
| `style` | CSS・見た目のみの変更（機能変更なし） |
| `refactor` | 動作を変えないコード改善 |
| `perf` | パフォーマンス改善 |
| `docs` | ドキュメントのみの変更 |
| `chore` | コンテンツ更新・設定変更・依存関係更新 |
| `test` | テストの追加・修正 |
| `build` | ビルド設定・スクリプトの変更 |
| `ci` | CI/CD 設定の変更 |

### Step 3: 破壊的変更を検出する

以下に該当する場合は `!` suffix を付ける（例: `feat!(): ...`）:
- 既存の URL が変わる
- `?from=card` クエリパラメータの動作が変わる
- 公開 API の互換性が壊れる
- コンポーネントの props インターフェースが変わる

### Step 4: 日本語でコミットメッセージを生成する

- 動詞から始める（「追加」「修正」「更新」「削除」「実装」「変更」など）
- 「何を」「どうした」が明確になるよう記述する
- 50文字以内を目安にする

### Step 5: ファイルをステージングする

`git add -A` より個別指定を優先する：

```bash
git add src/app/page.tsx src/app/globals.css
```

機密ファイル（`.env`、credentials 等）は絶対に含めない。

### Step 6: コミットを作成する

heredoc を使用してコミットを作成する：

```bash
git commit -m "$(cat <<'EOF'
feat(): 新機能の説明

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
EOF
)"
```

### Step 7: 結果を確認する

```bash
git log --oneline -3
```

フォーマットが `type(): 日本語説明` になっていることを確認する。

## 安全プロトコル

- `--force` push は絶対に行わない
- `--no-verify` でフックをスキップしない
- フック失敗時は問題を修正してから**新規コミット**を作成する（`--amend` は使わない）
- コミット前にユーザーへ確認する（自動コミットしない）
